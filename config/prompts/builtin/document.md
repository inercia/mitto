<investigate_before_answering>
Before updating documentation, read the code changes we made and the existing
user-facing docs. Understand what changed from the user's perspective before
writing or updating documentation.
</investigate_before_answering>

<task>
Update the user-facing documentation to reflect the changes we just made.
</task>

<instructions>

### Find and update:

1. **New features**: Add usage instructions, examples, configuration options
2. **Changed behavior**: Update existing docs to reflect new behavior
3. **New commands/options**: Document CLI flags, arguments, environment variables
4. **Configuration changes**: Update config file examples and explanations
5. **Breaking changes**: Clearly document migration steps

### Guidelines:

- Write for end users, not developers
- Include practical examples so users can copy and adapt them
- Keep language clear and concise
- Follow the existing documentation style and structure
- Update any table of contents or index if adding sections
- Test any commands or code snippets you include to ensure they work

</instructions>
